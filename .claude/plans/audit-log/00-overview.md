# Audit Log Module — Overview

## Purpose

Every write action across every module is automatically recorded — who did it, what changed,
when, and from where. Read-only in the UI. Access is gated by permissions like every other module:
if you have `audit-log.view-any` you can see the log; if not, you are blocked.

```
Admin creates Customer "John Smith"
  → activity_log row: event=created, subject=Customer#5, causer=User#2 (admin@sale-pro.test)
    properties: { attributes: { name: "John Smith", email: "...", status: "lead" } }

Admin updates Customer status "lead" → "active"
  → activity_log row: event=updated, subject=Customer#5, causer=User#2
    properties: { old: { status: "lead" }, attributes: { status: "active" } }

User logs in
  → activity_log row: log_name=auth, description=login, causer=User#2
    properties: { ip: "127.0.0.1" }
```

---

## Package

```bash
composer require spatie/laravel-activitylog
```

Provides:
- `activity_log` database table (via `php artisan activitylog:table`)
- `LogsActivity` trait — add to any Eloquent model for automatic logging
- `Activity` Eloquent model for reading log entries
- `activity()` helper for manual log entries (used for auth events)

---

## What Gets Logged

### Eloquent events (automatic via trait)

| Event | Trigger |
|-------|---------|
| `created` | Model saved for the first time |
| `updated` | Model attributes changed |
| `deleted` | Model soft-deleted or hard-deleted |
| `restored` | Soft-deleted model restored |

Models covered: `User`, `Customer`, `Department`, `Product`, `ProductListing`, `ProductCategory`

### Auth events (manual via listener)

| Description | Trigger |
|-------------|---------|
| `login` | User successfully authenticated |
| `logout` | User logged out |
| `login-failed` | Authentication attempt failed |

---

## Relationship to Other Modules

```
User      ──→ activity_log (as causer)
Customer  ──→ activity_log (as subject)
Department ─→ activity_log (as subject)
Product   ──→ activity_log (as subject)
ProductListing → activity_log (as subject)
ProductCategory → activity_log (as subject)
```

No foreign key constraints — uses Laravel morphable columns (`subject_type` / `subject_id`).
Records persist even if the subject model is deleted (subject becomes unreachable, not deleted).

## Implementation Order

1. `composer require spatie/laravel-activitylog`
2. `php artisan activitylog:table` → `php artisan migrate`
3. Publish config: `php artisan vendor:publish --provider="Spatie\Activitylog\ActivitylogServiceProvider" --tag="activitylog-config"`
4. Add `LogsActivity` to all models (`02-model-integration.md`)
5. Create `LogAuthActivity` listener + register in `AppServiceProvider` (`03-auth-events.md`)
6. Add Permission constants to `Permission` enum (`05-policy-permissions.md`)
7. Create `AuditLogPolicy` + register in `AppServiceProvider` (`05-policy-permissions.md`)
8. Create `AuditLogService` + `AuditLogController` (`04-controller.md`)
9. Create views (`06-views.md`)
10. Add routes (`08-seeders-routes.md`)
11. Create `AuditLogPermissionSeeder` + call from `DatabaseSeeder` (`08-seeders-routes.md`)
12. Tests (`07-tests.md`)

## Key Design Decisions

| Decision | Why |
|----------|-----|
| `logOnlyDirty()` | Only record attributes that actually changed — no noise on updates |
| `dontSubmitEmptyLogs()` | Skip saves that change nothing (e.g. touch() calls) |
| `logFillable()` | Log exactly the fields the app cares about — not internal timestamps |
| No FK on subject/causer | Logs persist after model deletion — history is immutable |
| Auth events via listener | Package doesn't auto-log auth — listener gives full control over format |
| Read-only UI | Audit log is append-only by design — no edit or delete from UI |
| 2 permissions only | View-any (list) + view (detail) — no create/edit/delete permissions needed |

## Key Rules

- `strict_types=1` on every PHP file
- `$this->authorize()` on every controller action
- Controller reads via service — never query `Activity` directly in controller
- Eager load `causer` and `subject` — never lazy load
- Every controller action has a Pest feature test
- Every service method has a Pest unit test
