# Product Module — E2E Tests

## STRICT RULE — READ FIRST
**The e2e-runner agent MUST:**
- Run E2E tests only
- Report pass/fail results for every journey and test case
- Report errors, screenshots, and failure details

**The e2e-runner agent MUST NOT:**
- Edit, create, or delete any application source files
- Fix bugs found during testing
- Modify controllers, views, models, services, routes, or migrations
- Touch anything outside `tests/E2E/`

If a test fails, **report it** — do not fix it. The developer reviews the report and decides what to fix.

---

## File
`tests/E2E/ProductTest.php` (Playwright via e2e-runner agent)

## Why E2E Over Feature Tests Alone
Pest feature tests pass data directly to the controller — they cannot catch:
- HTML form fields missing (the SKU edit bug: field not submitted because `<p>` not `<input>`)
- JavaScript `confirm()` dialogs on delete
- Visual rendering (sale price strike-through, badges, empty states)
- CSS-driven uppercase display vs actual stored value
- Filter bar interaction (combined filters, clear links)
- Flash message rendering after redirect

## Journey 3 — Create Product

### 3.1 Create form loads
- Click "New Product" button
- Assert URL is `/admin/products/create`
- Assert form has SKU, Name, Regular Price fields
- Assert Active checkbox is checked by default

### 3.2 Successful create
- Fill SKU: `test-001` (lowercase)
- Fill Name: `Test Product`
- Fill Regular Price: `29.99`
- Submit
- Assert redirect to `/admin/products/{id}`
- Assert success flash message visible
- Assert SKU displayed as `TEST-001` (uppercased)
- Assert `assertDatabaseHas('products', ['sku' => 'TEST-001'])`

### 3.3 SKU auto-uppercase (visual)
- Type `abc-123` in SKU field
- Assert displayed as `ABC-123` (CSS `text-transform: uppercase` on input)

### 3.4 Required fields validation
- Submit empty form
- Assert stay on create page
- Assert error messages for: SKU, Name, Regular Price

### 3.5 SKU uniqueness validation
- Seed product with SKU `TAKEN-001`
- Submit create form with SKU `TAKEN-001`
- Assert error "The sku has already been taken" (or similar)
- Assert form values repopulated (name, price still filled)

### 3.6 SKU uniqueness case-insensitive
- Seed product with SKU `TAKEN-001`
- Submit create form with SKU `taken-001` (lowercase)
- Assert uniqueness error shown (prepareForValidation uppercases before unique rule)

### 3.7 SKU regex validation
- Submit with SKU `INVALID SKU!` (has space and exclamation)
- Assert error "SKU may only contain letters, numbers, hyphens, and dots."

### 3.8 Sale price < regular price validation
- Fill Regular Price: `10.00`, Sale Price: `15.00`
- Submit
- Assert error "Sale price must be less than the regular price."

### 3.9 Sale price equal to regular price
- Fill Regular Price: `10.00`, Sale Price: `10.00`
- Submit
- Assert validation error (must be less than, not equal)

### 3.10 Category dropdown
- Seed categories: "Electronics", "Clothing"
- Open category dropdown
- Assert both categories visible + "— Uncategorised —" option
- Select "Electronics" → submit → assert product saved with correct category

### 3.11 Create without category (uncategorised)
- Leave category as "— Uncategorised —"
- Submit valid form
- Assert product created with null category_id
- Show page displays "Uncategorised"

### 3.12 Create inactive product
- Uncheck Active checkbox
- Submit
- Assert product created with `is_active = false`
- Show page shows "Inactive" badge

### 3.13 Values repopulate after validation failure
- Fill Name: "My Product", Regular Price: `10.00`, leave SKU empty

## Implementation Notes

- Seed via `php artisan db:seed --class=RoleSeeder` + `ProductPermissionSeeder` before suite
- Use `$page->waitForLoadState('networkidle')` after form submits
- For confirm dialogs: `$page->on('dialog', fn($d) => $d->accept())` to auto-accept
- For cancel dialog test: `$page->on('dialog', fn($d) => $d->dismiss())`
- Screenshot on failure for all journeys
